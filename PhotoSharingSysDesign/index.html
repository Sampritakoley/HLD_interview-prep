01 / Understanding the problem

What is this system, really?

Instagram is a photo and video sharing platform. Users upload photos, follow other users, and see a personalized feed of content from people they follow. The core challenge isn't the feature set — it's doing this for billions of users with sub-200ms latency.

Core features in scope

Photo Upload

Store, compress, and globally distribute photos within seconds of capture.

Feed Generation

Show posts from followed users, ranked and fast — even for accounts following thousands of people.

Social Graph

Follow / unfollow relationships that determine whose content you see.

Search & Discover

Find users and hashtags, surface trending content near real-time.

What makes this hard at scale

  • Read-heavy imbalance — roughly 100 reads for every 1 write. The system must be designed around reads, not writes.
  • The celebrity problem — accounts with 100M+ followers break naive fan-out models entirely.
  • Storage at scale — hundreds of terabytes uploaded per day, requiring tiered storage and CDN.
  • Feed freshness vs. speed — feeds must feel instant but also stay reasonably fresh.
02 / Requirements

Functional vs. non-functional requirements

Non-functional requirements are what actually drive architecture decisions — they're listed first for a reason.

Functional

  • Upload photos / videos
  • Follow / unfollow users
  • View personalized home feed
  • Like and comment on posts
  • Search users and hashtags
  • Global, fast content delivery
  • Notifications on likes / follows

Non-functional

  • 100M DAU, 2B total users
  • Feed load < 200ms (p99)
  • Photo upload < 2 seconds
  • 99.99% availability
  • Eventual consistency OK for feeds
  • Strong consistency for follows / likes
  • Durable storage — photos kept for years

Clarifying questions worth asking first

  • Should we support video, or just photos?
  • What's the read:write ratio?
  • Do celebrity accounts need special handling in the feed?
  • Is the feed ranked algorithmically, or purely chronological?
  • What's the geographic distribution — global, or one region?
  • Do we need Stories (ephemeral 24-hour content)?
  • What consistency model is acceptable for the feed?
03 / Estimation

Back-of-envelope math

Order-of-magnitude accuracy is the goal here, not precision. Every assumption is stated explicitly.

Assumptions

DAU                      = 100M users
Uploads per DAU          = 2 photos/day        → 200M uploads/day
Reads per DAU            = 200 views/day        → 20B reads/day
Avg photo size (compressed) = 200 KB
Avg follows per user     = 300

QPS

Write QPS:
  200M / 86,400 sec/day   = ~2,300 writes/sec
  Peak (3x average)       = ~7,000 writes/sec

Read QPS:
  20B / 86,400            = ~231,000 reads/sec
  Peak (3x average)       = ~700,000 reads/sec

Read:Write ratio ≈ 100:1 → design for read scalability above all else

Storage

Photos/day      = 200M x 200KB        = 40 TB/day
Metadata/day    = 200M rows x 500B    = ~100 GB/day
5-yr total      = 40TB x 365 x 5      = ~73 PB
With 3 resolutions stored             = ~220 PB over 5 years

Bandwidth

Upload bandwidth    = 7,000/s  x 200KB = 1.4 GB/sec
Download bandwidth  = 700,000/s x 200KB = 140 GB/sec
CDN offload (90% hit rate)  → only 14 GB/sec reaches origin

Component rationale

ComponentWhy it's there
CDNPhotos are static content. Caches at 200+ edge locations, cutting latency from ~300ms to ~20ms and absorbing 90%+ of read traffic.
API Gateway / LBSingle entry point for SSL termination, auth, rate limiting, and routing to the right service.
Upload ServiceStores the raw file to S3, triggers async processing, writes metadata, publishes a Kafka event. Never blocks on processing.
KafkaDecouples upload from fan-out. Upload returns success immediately; everything downstream happens async.
Feed Fanout WorkersConsume Kafka events and push new post IDs into followers' Redis feed lists.
Redis (feed cache)Pre-computed feeds as sorted lists — a sub-millisecond read instead of an expensive JOIN.
S3 / GCSPhotos are binary blobs. Object stores give near-infinite scale and 11 nines of durability.
PostgreSQL + CassandraPostgreSQL for users/follows (needs ACID). Cassandra for posts/likes (high write throughput, eventual consistency OK).

End-to-end data flow — photo upload

  1. User selects a photo; app requests a pre-signed S3 URL from Upload Service
  2. App uploads the photo directly to S3, bypassing our servers entirely
  3. App notifies Upload Service the upload is complete
  4. Upload Service writes metadata to Cassandra (status: processing), publishes to Kafka topic new_photo
  5. Photo Processing Worker generates thumbnail / medium / full resolutions, runs content moderation, marks status: published, warms the CDN
  6. Feed Fanout Worker fetches the uploader's follower list and pushes the new post ID into each follower's Redis feed list
  7. Follower opens the app → Feed Service reads from Redis (<1ms) → photo served via CDN

Total: "Posted!" appears in <500ms. Post is visible to followers in ~5–10 seconds (eventual consistency).

05 / Database design

Storage layer

Different data, different access patterns, different databases — not one engine forced to do everything.

DataDatabaseWhy
Users, follow graphPostgreSQLACID guarantees, strong consistency, moderate write volume
Posts, likes, commentsCassandraMassive write throughput, time-series access, eventual consistency OK
Feed cacheRedisSub-ms reads, pre-computed, rebuildable from source of truth
Photo / video filesS3 + CDNObject store built for large binary blobs

Posts table (Cassandra)

CREATE TABLE posts (
  user_id   UUID,
  post_id   TIMEUUID,
  photo_url TEXT,
  caption   TEXT,
  like_count INT,
  PRIMARY KEY (user_id, post_id)
) WITH CLUSTERING ORDER BY (post_id DESC);

-- Query: latest 20 posts by a user — single partition, sorted by time, free
SELECT * FROM posts WHERE user_id = 'user_123' LIMIT 20;

Partition key user_id keeps all of one user's posts on the same node. Clustering key post_id as a TIMEUUID gives free time-ordering with no ORDER BY cost.

Follows table (PostgreSQL)

CREATE TABLE follows (
  follower_id UUID REFERENCES users(user_id),
  followee_id UUID REFERENCES users(user_id),
  created_at  TIMESTAMPTZ DEFAULT NOW(),
  PRIMARY KEY (follower_id, followee_id)
);
CREATE INDEX idx_follows_followee ON follows(followee_id);

Sharding strategy

  • Posts — shard by user_id; all of a user's posts live on one shard
  • Follows — shard by follower_id, optimized for "who do I follow?" (the feed-critical query)
  • Photos (S3) — no manual sharding; AWS distributes internally
  • Consistent hashing for shard routing — adding shards moves only K/N keys, not a full reshuffle
06 / Deep dives

The three hardest problems

This is where architecture gets tested against failure modes, not just happy paths.

01 Feed generation at scale

When you open Instagram, we show posts from everyone you follow — ranked, in under 200ms. Following 1,000 people makes this nontrivial.

Naive — fan-in at read time
SELECT posts.* FROM posts
JOIN follows ON posts.user_id = follows.followee_id
WHERE follows.follower_id = 'current_user'
ORDER BY created_at DESC LIMIT 20;

JOIN across millions of rows = 500ms+ per query. At 700K reads/sec, this takes the database down.

Real solution — fan-out on write (push model)

On upload: fetch the uploader's follower list, push post_id into each follower's Redis list. On read: a single LRANGE call returns 20 IDs in under a millisecond.

The celebrity problem

A user with 400M followers needs 400M Redis writes per post. At 100K writes/sec, that's over an hour of lag.

Hybrid solution

Accounts above ~1M followers skip fan-out entirely. At read time, merge the pre-computed Redis feed with a fresh pull of recent posts from followed celebrities, then cache the merged result for ~5 minutes.

02 Photo upload pipeline

A 5MB photo on a slow connection still needs to feel instant to the user, while being processed into multiple resolutions reliably.

Naive

Client uploads to our server → server processes → stores to S3 → returns success. Processing takes 2–3 seconds and overwhelms a single service tier at 7,000 uploads/sec.

Real solution

Pre-signed S3 URL lets the client upload directly to S3. Our service only handles metadata. Processing — resize, compress, moderate — happens asynchronously via Kafka. Result: "Posted!" in <500ms.

03 Like counter accuracy vs. speed

A viral post can take 800 likes/sec — all hitting one row.

Naive
UPDATE posts SET like_count = like_count + 1 WHERE post_id = 'xyz';

Creates a hot row with lock contention and race conditions at scale.

Real solution

INCR in Redis for the count, SADD for dedup — both atomic, both instant. A background worker batch-writes the count to Cassandra every 30 seconds. Viral posts get the counter sharded across N Redis keys to avoid a hot key.

07 / Tradeoffs & alternatives

What we didn't choose, and what breaks next

Rejected alternatives

AlternativeWhy rejected
MySQL for postsCan't sustain 7K writes/sec without expensive sharding; doesn't need ACID here
Elasticsearch for feedNot built for write-heavy workloads at this scale — reserved for search instead
Pure push modelBreaks for celebrities — hours of fan-out lag
On-the-fly resizingToo expensive at 700K reads/sec — pre-generate at upload time
WebSockets everywhereStateful, harder to scale — reserved for real-time notifications only

What breaks at 10x scale

ProblemFix
Redis memory exhaustionOnly cache feeds for users active in the last 30 days; recompute on first login otherwise
Cassandra write hotspotRedis write buffer + batched persistence + counter sharding
S3 egress costsAggressive CDN caching, targeting 99% cache hit rate
Kafka lag on viral fan-outAuto-scaling consumer groups; de-prioritize celebrity fan-out vs. regular users
08 / Summary

Cheat sheet

The ten things worth remembering before walking into the room.

Key numbers

100M DAU · 7K write QPS · 700K read QPS · 40TB/day · Read:Write 100:1

Key databases

PostgreSQL (users/follows) · Cassandra (posts/likes) · Redis (feed/counters) · S3 (photo blobs)

Feed strategy

Push for regular users, pull-at-read for celebrities, merged at read time

Photo pipeline

Pre-signed S3 URL → direct upload → async processing → CDN distribution

Like counter

Redis INCR + SADD instantly → batched persist to Cassandra every 30s

Hardest part

The celebrity fan-out problem — hybrid push/pull is the key insight

Backbone

Kafka decouples upload from fan-out, notifications, and processing

Non-negotiable

CDN — 90%+ of reads must be served from edge, not origin

Sharding keys

Posts by user_id · Follows by follower_id · consistent hashing for rebalancing

Consistency model

Eventual for feed/likes, strong for follows — explicit tradeoff, not an accident
