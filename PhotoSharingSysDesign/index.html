01 / Understanding the problem

What is this system, really?

Instagram is a photo and video sharing platform. Users upload photos, follow other users, and see a personalized feed of content from people they follow. The core challenge isn't the feature set — it's doing this for billions of users with sub-200ms latency.

Core features in scope

Photo Upload

Store, compress, and globally distribute photos within seconds of capture.

Feed Generation

Show posts from followed users, ranked and fast — even for accounts following thousands of people.

Social Graph

Follow / unfollow relationships that determine whose content you see.

Search & Discover

Find users and hashtags, surface trending content near real-time.

What makes this hard at scale

  • Read-heavy imbalance — roughly 100 reads for every 1 write. The system must be designed around reads, not writes.
  • The celebrity problem — accounts with 100M+ followers break naive fan-out models entirely.
  • Storage at scale — hundreds of terabytes uploaded per day, requiring tiered storage and CDN.
  • Feed freshness vs. speed — feeds must feel instant but also stay reasonably fresh.
02 / Requirements

Functional vs. non-functional requirements

Non-functional requirements are what actually drive architecture decisions — they're listed first for a reason.

Functional

  • Upload photos / videos
  • Follow / unfollow users
  • View personalized home feed
  • Like and comment on posts
  • Search users and hashtags
  • Global, fast content delivery
  • Notifications on likes / follows

Non-functional

  • 100M DAU, 2B total users
  • Feed load < 200ms (p99)
  • Photo upload < 2 seconds
  • 99.99% availability
  • Eventual consistency OK for feeds
  • Strong consistency for follows / likes
  • Durable storage — photos kept for years

Clarifying questions worth asking first

  • Should we support video, or just photos?
  • What's the read:write ratio?
  • Do celebrity accounts need special handling in the feed?
  • Is the feed ranked algorithmically, or purely chronological?
  • What's the geographic distribution — global, or one region?
  • Do we need Stories (ephemeral 24-hour content)?
  • What consistency model is acceptable for the feed?
03 / Estimation

Back-of-envelope math

Order-of-magnitude accuracy is the goal here, not precision. Every assumption is stated explicitly.

Assumptions

DAU                      = 100M users
Uploads per DAU          = 2 photos/day        → 200M uploads/day
Reads per DAU            = 200 views/day        → 20B reads/day
Avg photo size (compressed) = 200 KB
Avg follows per user     = 300

QPS

Write QPS:
  200M / 86,400 sec/day   = ~2,300 writes/sec
  Peak (3x average)       = ~7,000 writes/sec

Read QPS:
  20B / 86,400            = ~231,000 reads/sec
  Peak (3x average)       = ~700,000 reads/sec

Read:Write ratio ≈ 100:1 → design for read scalability above all else

Storage

Photos/day      = 200M x 200KB        = 40 TB/day
Metadata/day    = 200M rows x 500B    = ~100 GB/day
5-yr total      = 40TB x 365 x 5      = ~73 PB
With 3 resolutions stored             = ~220 PB over 5 years

Bandwidth

Upload bandwidth    = 7,000/s  x 200KB = 1.4 GB/sec
Download bandwidth  = 700,000/s x 200KB = 140 GB/sec
CDN offload (90% hit rate)  → only 14 GB/sec reaches origin

Component rationale

ComponentWhy it's there
CDNPhotos are static content. Caches at 200+ edge locations, cutting latency from ~300ms to ~20ms and absorbing 90%+ of read traffic.
API Gateway / LBSingle entry point for SSL termination, auth, rate limiting, and routing to the right service.
Upload ServiceStores the raw file to S3, triggers async processing, writes metadata, publishes a Kafka event. Never blocks on processing.
KafkaDecouples upload from fan-out. Upload returns success immediately; everything downstream happens async.
Feed Fanout WorkersConsume Kafka events and push new post IDs into followers' Redis feed lists.
Redis (feed cache)Pre-computed feeds as sorted lists — a sub-millisecond read instead of an expensive JOIN.
S3 / GCSPhotos are binary blobs. Object stores give near-infinite scale and 11 nines of durability.
PostgreSQL + CassandraPostgreSQL for users/follows (needs ACID). Cassandra for posts/likes (high write throughput, eventual consistency OK).

End-to-end data flow — photo upload

  1. User selects a photo; app requests a pre-signed S3 URL from Upload Service
  2. App uploads the photo directly to S3, bypassing our servers entirely
  3. App notifies Upload Service the upload is complete
  4. Upload Service writes metadata to Cassandra (status: processing), publishes to Kafka topic new_photo
  5. Photo Processing Worker generates thumbnail / medium / full resolutions, runs content moderation, marks status: published, warms the CDN
  6. Feed Fanout Worker fetches the uploader's follower list and pushes the new post ID into each follower's Redis feed list
  7. Follower opens the app → Feed Service reads from Redis (<1ms) → photo served via CDN

Total: "Posted!" appears in <500ms. Post is visible to followers in ~5–10 seconds (eventual consistency).

05 / Database design

Storage layer

Different data, different access patterns, different databases — not one engine forced to do everything.

DataDatabaseWhy
Users, follow graphPostgreSQLACID guarantees, strong consistency, moderate write volume
Posts, likes, commentsCassandraMassive write throughput, time-series access, eventual consistency OK
Feed cacheRedisSub-ms reads, pre-computed, rebuildable from source of truth
Photo / video filesS3 + CDNObject store built for large binary blobs

Posts table (Cassandra)

CREATE TABLE posts (
  user_id   UUID,
  post_id   TIMEUUID,
  photo_url TEXT,
  caption   TEXT,
  like_count INT,
  PRIMARY KEY (user_id, post_id)
) WITH CLUSTERING ORDER BY (post_id DESC);

-- Query: latest 20 posts by a user — single partition, sorted by time, free
SELECT * FROM posts WHERE user_id = 'user_123' LIMIT 20;

Partition key user_id keeps all of one user's posts on the same node. Clustering key post_id as a TIMEUUID gives free time-ordering with no ORDER BY cost.

Follows table (PostgreSQL)

CREATE TABLE follows (
  follower_id UUID REFERENCES users(user_id),
  followee_id UUID REFERENCES users(user_id),
  created_at  TIMESTAMPTZ DEFAULT NOW(),
  PRIMARY KEY (follower_id, followee_id)
);
CREATE INDEX idx_follows_followee ON follows(followee_id);

Sharding strategy

  • Posts — shard by user_id; all of a user's posts live on one shard
  • Follows — shard by follower_id, optimized for "who do I follow?" (the feed-critical query)
  • Photos (S3) — no manual sharding; AWS distributes internally
  • Consistent hashing for shard routing — adding shards moves only K/N keys, not a full reshuffle
06 / Deep dives

Deep dive on 3 critical components

For each: the problem, the naive solution, why it fails, and the real solution used at scale.

01 Feed generation at scale

The problem: When you open Instagram, we need to show you posts from everyone you follow, ranked by relevance, in under 200ms. You might follow 1,000 people. That means scanning 1,000 users' recent posts, ranking them, and returning the top 20.

Naive solution — Pull model (fan-in at read time)
When user opens feed:
  SELECT posts.* FROM posts
  JOIN follows ON posts.user_id = follows.followee_id
  WHERE follows.follower_id = 'current_user'
  ORDER BY created_at DESC LIMIT 20;

Why it fails: JOIN across millions of rows = 500ms+ query. If you follow 1,000 people, that's 1,000 posts to rank. 700K reads/sec × 500ms means the database gets destroyed.

Real solution — Push model (fan-out on write)
When user A uploads a photo:
  1. Fetch A's follower list (say, 500 followers)
  2. For each follower, append photo_id to their feed in Redis:
     LPUSH feed:user_B photo_id_xyz
     LTRIM feed:user_B 0 999   (keep only 1000 most recent)
  3. This happens asynchronously via Kafka
 
When user B opens their feed:
  LRANGE feed:user_B 0 19   → returns 20 post IDs in <1ms
  Fetch post details by ID from Cassandra (parallel)
  Fetch photos from CDN
 
Read time: <50ms total ✓
But — the celebrity problem

Taylor Swift has 400M followers. When she posts, we'd need to update 400M Redis keys. At 100K writes/sec, that takes 4,000 seconds — over an hour. Totally unacceptable.

Hybrid solution (what Instagram actually uses)
Define: "celebrity" = user with > 1M followers
 
For regular users (< 1M followers):
  → Use push model (fan-out on write to Redis)
 
For celebrities (> 1M followers):
  → Skip the fan-out. Store their posts separately.
  → At READ TIME, merge:
     1. Read follower's pre-computed feed from Redis (regular accounts)
     2. Fetch latest 20 posts from each celebrity they follow
     3. Merge + rank the two lists
     4. Cache this merged result for 5 minutes

This means 99.9% of users get instant feed from Redis, and the extra celebrity merge adds only ~10ms.

02 Photo upload pipeline

The problem: A user uploads a 5MB JPEG from a 3G connection. We need to store it durably, generate 3 resolutions, and make it available globally in seconds — without the user waiting.

Naive solution
User → Upload Service → process image → store S3 → return success

Why it fails: Processing (resize, compress, filter) takes 2–3 seconds. The user waits on a slow mobile connection. If processing fails, the upload fails. 7,000 uploads/sec overwhelms a single service.

Real solution — async pipeline with direct S3 upload
Step 1: Pre-signed URL (eliminate upload bottleneck)
  Client requests: GET /upload/presign
  Server returns: {presigned_url, post_id}
  Client uploads DIRECTLY to S3 — bypasses our servers entirely!
 
Step 2: Client notifies our service that upload is done
  POST /photos {post_id, caption}
  Service writes to DB with status = "processing"
  Publishes to Kafka: {post_id, s3_key, user_id}
 
Step 3: Photo Processing Workers (async)
  - Download from S3
  - Generate: thumbnail (150x150), medium (720px), original
  - Run content moderation (NSFW detection via ML model)
  - Store 3 versions back to S3
  - Update DB status = "published"
  - Invalidate/warm CDN
 
Step 4: Kafka triggers Feed Fanout Worker (separate)

Result: user sees "Posted!" in <500ms. Photo appears in followers' feeds in ~5–10 sec.

03 Like count accuracy vs. speed

The problem: A viral post gets 50,000 likes in 1 minute — that's 800 writes/sec to a single row. The heart must fill instantly when you like a post, but the global counter update must not block your experience.

Naive solution
UPDATE posts SET like_count = like_count + 1 WHERE post_id = 'xyz';
INSERT INTO likes (user_id, post_id) VALUES (...);

Why it fails: Row-level locking on a single row at 800 writes/sec creates a "hot row" — all writes queue up on one DB row. Counters can become wrong with race conditions.

Real solution — counter sharding + Redis
Step 1: User likes a post
  → Write to Redis immediately:
     INCR like_count:post_xyz          (atomic, returns new count)
     SADD liked_by:post_xyz user_123   (track who liked)
  → Return success to user instantly (<1ms)
 
Step 2: Write-behind to DB (async, batched)
  Every 30 seconds, a worker:
  - Reads current count from Redis
  - Batch-writes to Cassandra:
    UPDATE post_metrics SET like_count = {redis_val} WHERE post_id = 'xyz'
 
Step 3: Counter sharding for viral posts
  Instead of 1 Redis key:
    like_count:post_xyz:shard_0
    like_count:post_xyz:shard_1
    ...
    like_count:post_xyz:shard_9
 
  Random writes to one of 10 shards (eliminates hot key)
  Read time: SUM all 10 shards (still fast)

Result: like feel is instant, counter accuracy is ±5% (acceptable — "4.2M likes" not "4,193,847"), DB load is 99% reduced, and no duplicate likes since Redis SET enforces uniqueness.

07 / Tradeoffs & alternatives

What we did NOT choose and why

Rejected alternatives

AlternativeWhy rejected
MySQL instead of Cassandra for postsMySQL is ACID, which we don't need for posts. Can't handle 7K writes/sec without expensive sharding setup. Cassandra's eventual consistency fits social media posts perfectly.
Elasticsearch for feed generationTempting because it supports ranking/scoring, but it's not designed for write-heavy workloads at this scale and has complex operational overhead. We use it only for search (hashtags, usernames), not feed.
Pure push model for everyoneBreaks for celebrities (400M fanout = hours of lag). Our hybrid approach handles 99% of users with push, and 1% celebrities with pull-at-read-time.
On-the-fly image resizing (served by server)Services like Imgix do this — but at 700K reads/sec, even a 5ms resize operation = 3,500 CPU cores busy. Pre-generating 3 sizes at upload time is far cheaper operationally.
WebSockets for all API callsWebSockets are stateful, making horizontal scaling harder. We use WebSockets only for real-time notifications (likes, follows). Regular feed/photo API uses REST + HTTP/2, which is stateless and simpler to scale.

What breaks at 10× scale and how we fix it

Problem — Redis memory exhaustion

At 10× DAU = 1B users × 1000 post IDs × 8 bytes = 8TB RAM. Too expensive.

Fix

Only keep feeds in Redis for active users (logged in within 30 days). Inactive users get feeds recomputed on first login. Also compress post IDs.

Problem — Cassandra write hotspot (viral posts)

A post with 100M likes all writing to one Cassandra partition.

Fix

Write to Redis counters only. Persist to Cassandra with write batching every 60 seconds. Counter sharding eliminates the Redis hot key too.

Problem — S3 egress costs at 10×

140 GB/sec × 10 = 1.4 TB/sec download = astronomical egress fees.

Fix

Aggressive CDN caching (cache-control: max-age=31536000 for photos, since they never change). A 99% cache hit rate means only 14 GB/sec hits S3.

Problem — Kafka lag during viral events

A single celebrity post causes 50M fanout events flooding Kafka.

Fix

Auto-scaling consumer groups. Priority queues — regular users get fanout first, celebrity fanout de-prioritized (users accept a slight delay for celebrity posts).
